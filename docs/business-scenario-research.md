# CleanCare 清洁电器客服业务场景研究

## 1. 文档目的

本文说明 CleanCare Agent 为什么选择清洁电器作为 Agentic RAG 展示场景、公开电商与厂商服务页面提供了哪些可借鉴的信息结构，以及项目据此定义的用户、意图和知识文档体系。

边界声明：

- 本项目仅覆盖扫地机器人、空气净化器、净水器、加湿器四类清洁电器。
- T20、X20 Pro、R10、R20、P400、P500、W300、W500、H100、H200 均为 CleanCare 模拟型号。
- 商品参数、价格、库存、订单、保修和售后工单均为 mock 数据，不代表小米、追觅或石头的真实商品与系统。
- 本项目没有接入真实支付、物流、ERP、厂商账号或售后系统。

## 2. 为什么选择清洁电器

清洁电器同时具备“高密度静态知识”和“强时效动态数据”，适合完整展示 RAG 与 Agent 的边界。

1. 参数密度高。吸力、导航、续航、适用面积、噪声、耗材、越障和联网能力共同影响选购，单一关键词无法稳定回答组合问题。
2. 型号关系复杂。用户常比较相近型号，或通过“上次买的那个”“这个滤芯”引用历史实体，需要会话记忆和指代消解。
3. 动静态数据混合。规格、说明书和政策适合知识库检索；价格、库存、订单和保修必须通过受控工具查询。
4. 故障处理具有状态。诊断需要沿故障树逐步追问，且漏电、冒烟、焦味、进水、过热等场景必须立即停止 DIY 指导。
5. 售后判断依赖多个条件。订单日期、签收状态、商品使用情况、政策生效时间和例外条款必须联合判断，不能只检索一句“七天无理由”。
6. 展示效果可量化。可分别评估检索召回、证据忠实度、工具成功率、诊断轮数、Token 消耗和端到端延迟。

## 3. 公开页面观察

以下内容仅用于研究信息结构和客服任务，不复制厂商数据作为 CleanCare 产品事实。

### 3.1 小米商城与支持页面

小米商品页会同时展示吸力、导航、自动回充、续航、适用面积、机身尺寸和安全保护等信息；支持 FAQ 则按“如何连接 APP、如何重置 Wi-Fi、如何恢复出厂设置”等任务组织操作步骤。这说明商品详情与操作说明应采用不同分块策略，步骤不能被任意截断。

### 3.2 追觅产品与支持页面

追觅公开页面将机器人、洗地机、无线吸尘器等产品分类，并提供说明书、FAQ、使用视频和售后入口。公开隐私说明还表明智能清洁设备服务会涉及设备状态、耗材寿命、清洁记录和错误信息。对 CleanCare 的启发是：耗材查询、设备故障和购买记录属于不同数据域，工具必须最小授权并保留审计。

### 3.3 石头产品、说明书与售后页面

石头官网将产品介绍、服务支持、说明书和售后政策分开维护。说明书包含安装、联网、拖地、维护、安全限制和故障处理；售后页面独立呈现服务政策。对 CleanCare 的启发是：说明书应按操作任务分块，安全警告必须与步骤同块；售后政策应保存条款、适用条件、例外和生效时间。

## 4. 用户体系

| 用户类型 | 典型目标 | 主要风险 |
|---|---|---|
| 选购用户 | 查参数、比较型号、按预算和家庭条件推荐 | 只召回一个型号、忽略硬约束、使用过期价格 |
| 已购用户 | 查使用方法、耗材兼容、故障排查 | 指代错误、步骤过载、安全红线遗漏 |
| 售后用户 | 判断退换货、保修、创建工单 | 政策误用、订单事实缺失、未确认即执行写操作 |

## 5. 意图体系

| 一级意图 | 典型问题 | 处理模式 |
|---|---|---|
| `product_parameter` | “T20 吸力多少？” | 快速 RAG |
| `product_comparison` | “T20 和 X20 Pro 哪个适合养猫？” | 分型号检索 + 场景指南 + 对比生成 |
| `product_recommendation` | “120 平两只猫，预算 5000 推荐” | 约束提取 + 候选召回 + 硬条件过滤 + 价格库存 |
| `accessory_compatibility` | “上次买的净化器滤芯多少钱？” | 购买记录 + 主机定位 + 兼容关系 + 价格 |
| `usage_instruction` | “怎么重置 Wi-Fi？” | 任务式说明书检索 |
| `troubleshooting` | “充不进电怎么办？” | 故障树状态机 + 多轮澄清 |
| `after_sales` | “买了 20 天拆了还能退吗？” | 订单事实 + 当前政策 + 条件化判断 |
| `out_of_scope` | 非清洁电器或高风险越权请求 | 明确边界或转人工 |

多意图问题保留主次关系。例如“X20 Pro 适合地毯吗，现在多少钱”以参数/场景判断为主，同时调用价格工具；不能把两部分压成单一检索。

## 6. 知识文档体系

| 文档类型 | 内容单位 | 关键元数据 | 推荐分块方式 |
|---|---|---|---|
| `product_detail` | 单型号商品详情 | 品类、型号、版本 | 按章节分块，参数表保持完整 |
| `product_parameter` | 单型号结构化参数 | 型号、参数口径 | 每个型号一块 |
| `comparison` | 多型号同口径比较 | 型号集合、维度 | 按维度分块，保留全部型号 |
| `purchase_guide` | 场景与约束建议 | 户型、宠物、地毯 | 按用户场景分块 |
| `accessory_compatibility` | 主机与配件关系 | 主机型号、配件 SKU | 每条兼容关系一块 |
| `user_manual` | 一个操作任务 | 型号、任务、安全级别 | 步骤与警告不可拆散 |
| `troubleshooting` | 一个故障树节点 | 节点 ID、父节点、风险级别 | 每节点一块 |
| `after_sales_policy` | 条款及例外 | 生效时间、地区、品类 | 条款、条件、例外同块 |
| `faq` | 一问一答 | 型号、主题 | 一问一答一块 |

## 7. Agentic RAG 决策边界

系统不应让所有请求都进入 Agent 循环：

- 单型号、单事实、证据充分的问题走快速 RAG，降低延迟和 Token。
- 多型号、多约束、跨数据域或需要写操作的问题进入 Plan-and-Execute/ReAct。
- 故障诊断使用有状态 Skill，单轮最多推进一个高区分度问题。
- 创建售后工单属于有副作用操作，必须满足鉴权、超时、审计、用户确认和幂等键。
- ReAct 最多 5 步；重复 action 与相同参数必须拦截。

## 8. 项目实现与后续替换

当前知识条目统一使用 `mock://` 来源，目的是让评测结果可复现，并能在演示中清楚区分“已实现的工程链路”和“未来可替换的数据源”。替换为真实数据时，应保留现有文档类型、元数据字段、版本和生效时间，禁止直接覆盖历史政策或静默改变参数口径。
