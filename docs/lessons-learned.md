# CleanCare Agent 失败教训

## 修复前后总览

| 教训 | 修复前 | 修复后 |
|---|---|---|
| Few-Shot 过度具体 | 模型复制示例数字和型号 | 示例覆盖不同品类并强调证据优先 |
| 路由关键词过宽 | “换”同时命中换货和换滤芯 | 关键词结合实体、上下文和置信度 |
| Reflection 原策略重跑 | 重复召回同一批结果 | 每次 rerun 改变过滤或 query |
| Token 预算检查过晚 | 已生成长计划后才发现超预算 | 计划阶段限制步骤和高成本动作 |

## 教训一：Few-Shot 太具体会诱导复制

### 问题表现

早期对比 Prompt 只给出“T20 与 X20 Pro”的完整示例，模型在回答净化器对比时也倾向输出吸力、地毯和滚刷维度。

### 根因分析

示例同时承担格式和领域知识两个角色，模型把示例中的具体维度误当作固定模板。

### 解决方案

Prompt 示例跨扫地机、净化器和售后场景；输出模板只规定“同口径维度、证据、取舍”，具体维度必须由证据决定。代码位于 `internal/prompt/templates.go`。

### 验证结果

模板测试确保每个 Prompt 至少两个示例且能渲染。该问题已纳入 2026-06-13 的 200 条真实模型评测观察范围；若 Prompt 或工具链继续变更，应以新的当前版本回归为准，仍不声称已达到线上指标。

## 教训二：规则路由关键词不能过宽

### 问题表现

“滤芯怎么换”和“用了 20 天能换货吗”都包含“换”，简单包含匹配会把使用指导误判为售后。

### 根因分析

关键词缺少对象实体和动作搭配，也没有在低置信度时交给 LLM。

### 解决方案

RuleRouter 使用型号、订单号和领域动作组合打分；低置信度由 HybridRouter 调用 LLM，规则错误不直接覆盖高置信上下文。代码位于 `internal/intent/rule_router.go` 与 `internal/intent/hybrid_router.go`。

### 验证结果

现有规则单测覆盖价格、订单、保修、故障和越界；第二阶段继续增加多意图与 RouteTrace 测试。

## 教训三：Reflection 不改变策略只会浪费 Token

### 问题表现

检索质量低时直接用相同 query、相同 filter 和相同 TopK 重跑，结果几乎不变，却额外消耗一次检索和生成。

### 根因分析

Reflection 只给出“重跑”动作，没有把失败原因转换成检索策略。

### 解决方案

rerun 必须携带原因和策略：先使用改写 query，再去掉 doc_type filter，最后退化为 keyword-only。代码入口为 `internal/agent/agentic_runner.go` 的 rerun 分支和 `internal/agent/reflection.go`。

### 验证结果

已有 rerun 路径测试验证可重新检索；策略分级与效果增益将在第三阶段补充测试和 trace 字段。

## 教训四：Token 预算必须前移到规划阶段

### 问题表现

若只在每步执行前检查预算，LLM 可能先生成包含多次检索、多个工具和反思的长计划，随后大量步骤被跳过，计划本身已经浪费 Token。

### 根因分析

预算被当作运行时中止条件，而不是 Planner 的输入约束。

### 解决方案

`TokenBudget` 进入 PlanRequest；规划器限制总步骤不超过 5，并优先复用已知证据。预算接近上限时选择规则规划和抽取式生成。代码位于 `internal/agent/llm_planner.go` 与 `internal/agent/agentic_runner.go`。

### 验证结果

Runner 测试覆盖 Token 预算和最大步数。2026-06-13 的 200 条真实模型评测已记录运行级 Token；意图分类、查询改写、规划、生成、反思等阶段分布仍需在后续当前版本回归中补齐。
